### _"웹툰 작가의 경험으로 사용자 중심 서비스를 만들어갈 개발자, 정다희 입니다."_

저는 웹툰 작가로 약 7년간 활동하며 7일 단위의 마감 일정을 소화해냈으며, 작업을 세분화하고 우선순위를 관리하며 일정과 품질의 균형을 유지하는 책임감을 길렀습니다. 또한 편집자의 피드백 의도를 빠르게 파악하고 반영하여 불필요한 재수정을 줄이고 업무 효율을 높였습니다.

개발자 전향을 목표로 한 후, 팀 프로젝트를 통해 React와 Spring Boot 기반의 웹 서비스를 구축하고 JWT를 활용한 사용자 인증 로직을 구현했습니다. AWS EC2와 Docker를 활용해 서비스를 직접 배포하며 컨테이너 기반의 운영 환경을 구축했고, 기능 구현에 그치지 않고 개발부터 배포까지의 전 과정을 경험하며 서비스 운영에 대한 이해를 넓혔습니다.

새로운 기술을 빠르게 습득하고 실제 프로젝트에 검증하며 개선하는 것이 저의 강점입니다. 앞으로도 책임감 있는 일정 관리와 협업을 바탕으로 신뢰받는 개발자로 성장하겠습니다.

### 지원 동기, 어째서 개발인가?

작가들을 위한 리소스 거래 사이트, 작품 연재 플랫폼등을 이용해오며 사이트 사용자들에 대한 이해도 부족, 사용자의 경험을 이해하는 동시에 개발 역량을 갖춘 인력의 부재 등으로 불편함을 감수할 수 밖에 없었습니다. 이는 직접 서비스를 설계하고 구현하고 싶다는 목표로 이어졌고, React와 Spring Boot를 활용한 팀 프로젝트에서 요구사항 분석부터 구현, AWS 배포까지의 과정을 수행하며 실제 서비스 개발 경험을 쌓았습니다.

창작 활동을 통해 체득한 서비스 중심의 사고방식을 바탕으로, 앞으로도 '사용자에게 필요한 서비스'를 먼저 고민하고 이를 기술로 해결하는 개발자가 되고자 지원합니다.

### 관련 교육 이수

_**풀스택 역량 확보와 배포 환경 구축 및 협업 프로세스 경험**_

풀스택 웹 개발 교육 과정을 통해 Java, Spring Boot, React, TypeScript, MySQL을 활용한 웹 서비스 개발 전반을 학습하며 실무 프로젝트를 수행했습니다.&#x20;

최종 프로젝트에서는 React와 Spring Boot 기반의 웹 서비스를 구축하고 JWT를 활용한 사용자 인증 로직을 구현했습니다. 이후 AWS EC2와 Docker를 활용해 서비스를 직접 배포하며 컨테이너 기반의 운영 환경을 구축했고, 기능 구현에 그치지 않고 개발부터 배포까지의 전 과정을 경험하며 서비스 운영에 대한 이해를 넓혔습니다.

### 성격의 장·단점

_**리스크를 줄이는 습관과 꾸준한 자기관리**_

저의 가장 큰 장점은 잠재적인 리스크를 사전에 발견하고 대비하는 습관입니다. 팀 프로젝트 초기 단계에서 발생할 수 있는 소스코드 혼선을 방지하기 위해 개발 컨벤션과 폴더 구조를 제안했습니다. 초기에 공통 규칙을 정립한 덕분에 코드 충돌과 휴먼 에러를 줄여 보다 체계적으로 프로젝트를 수행할 수 있었습니다.

반면, 목표에 과도하게 몰입하는 경향이 있어 한때는 체력 관리를 소홀히 했던 경험도 있었습니다. 이후 곧 지속 가능한 성과를 내기 위해서는 자기관리가 필수적임을 깨달았습니다. 지금은 운동과 식단 관리를 병행하며 약 20kg을 감량했고, 주 2회 규칙적인 운동을 이어가고 있습니다. 이러한 습관은 장기 프로젝트에서도 꾸준한 집중력과 안정적인 일정 관리를 유지하는 기반이 되고 있습니다.

<mark style="color:blue;">**리스크 예방, 자기관리**</mark>

### 프로젝트를 마치며...

_**신뢰를 만드는 협업의 자세**_

프로젝트의 규모에 따라 기획부터 마감까지 전 과정을 책임지는 1인 제작부터, 역할을 분담하여 여러 구성원과 협업하는 팀 업무까지 다양한 업무를 수행해왔습니다. 그러한 경험은 개발 프로젝트에서도 이어졌습니다. 제가 맡은 기능은 책임감 있게 구현내는 동시에, 팀원들이 제 코드를 사용하기 쉽도록 가독성이 좋은 코드를 만들려 고민하였습니다.

무엇보다 프로젝트를 통해 가장 크게 성장한 부분은 소통의 자세였습니다. 의견이 다를 때에는 프로젝트의 완성도와 효율성을 우선으로 판단했고, 제 판단 역시 언제든 틀릴 수 있다는 전제 아래 서로의 의견을 검증하며 더 나은 해결책을 찾고자 했습니다. 실제로 로직 설계 과정에선 팀원이 제시한 방식이 더 코드가 간결하다는 점을 확인한 후 이를 수용하여 프로젝트를 진행한 경험이 있습니다. 또한 제 판단이 부족했던 부분은 솔직하게 인정하고 개선해 나갔습니다.&#x20;

앞으로도 열린 자세와 원활한 소통을 바탕으로 팀에 신뢰를 줄 수 있는 개발자로 성장하고자 합니다.

### 입사 후 포부

_**이해와 소통으로 가치를 만드는 개발자**_

저는 개발자가 단순히 코드만 작성하는 직무라고 생각하진 않습니다. 기술은 문제를 해결하기 위한 수단의 일종이며, 타인을 이해하고자 하는 관심과 소통이 합쳐질때 더 좋은 결과물을 만든다고 생각합니다. 입사 후에는 회사의 서비스와 개발 문화를 빠르게 이해하고, 기획 의도와 요구사항을 정확히 파악하는 개발자가 되겠습니다. 코드 리뷰와 피드백 역시 단순한 수정 사항이 아닌 더 나은 설계와 개발 방식을 배우는 과정으로 받아들이겠습니다.

또한 작은 기능 하나도 사용자 경험과 서비스의 안정성을 고려하는 책임감을 가지고, 기술과 신뢰를 함께 쌓아 궁극적으론 팀이 믿고 맡길 수 있는 개발자로 성장하겠습니다.
